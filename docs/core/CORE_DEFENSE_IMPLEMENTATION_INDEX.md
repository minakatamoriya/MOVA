# 守护核心模式实现入口索引

本文档用于把 docs/core 下的设计文档整理成一份更适合 AI 协助开发和实际实现排期理解的入口索引。

目标不是重复写设计内容，而是回答四个问题：

1. 某个系统的主定义在哪份文档
2. 该系统当前要先实现什么
3. 它依赖哪些前置系统
4. 实现顺序应该如何安排

## 4. 系统依赖关系

推荐按以下依赖理解：

1. 战场结构
2. 怪物推进与施压
3. 自动索敌
4. 金币产出与共享池
5. 玩家成长与核心成长入口
6. 核心组件原型
7. 精英掉落接入
8. 终局前台包装与长局调参

如果前面的依赖还没稳定，不建议直接深做后面的系统。

例如：

- 自动索敌没定，不要急着深做六职业基础技能细节。
- 金币系统没成，不要急着细抠五档成长数值。
- 核心组件边界没定，不要急着扩大量模块池。

---

## 5. 推荐首批原型覆盖范围

第一轮更推荐只覆盖以下原型组合：

1. 一个近战职业原型
2. 一个中程压场职业原型
3. 一个远程热点选区职业原型
4. 一个拦截核心组件
5. 一个灼烧或补伤核心组件
6. 一个护盾或回生核心组件

这样可以最快验证：

- 自动索敌是否成立
- 射程梯度是否成立
- 金币竞争是否成立
- 核心火力边界是否成立

## 6. AI 开发时的常见误区

后续如果由 AI 协助实现，当前最应避免以下误区：

1. 把金币系统理解成单纯数值表
   - 实际上金币系统同时承担节奏、站位、成长竞争和精英优先级设计。

2. 把自动索敌理解成最近敌人搜索
   - 实际上自动索敌应是区域热点、目标粘性和守家切换的组合状态机。

3. 把核心组件理解成另一套角色 DPS 树
   - 实际上核心组件只应提供近核防守、止损和条件反制。

4. 在首轮原型中平铺六职业完整体系
   - 实际上应先做代表职业覆盖不同机制，再扩完整职业池。

## 7. 当前建议的实现顺序

当前推荐的实际顺序是：

1. 稳定战场分层与怪潮推进
2. 锁定自动索敌第一版状态机
3. 跑通最小金币产出与共享池
4. 接入最小玩家 / 核心成长闭环
5. 做出首批职业基础原型
6. 做出首批核心组件原型
7. 接入精英高价值掉落
8. 再调终局前台包装与长局压力

如果出现“现在先做什么”的争论，默认回到这个顺序判断。
